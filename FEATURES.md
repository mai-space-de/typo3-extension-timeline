# mai_timeline — Feature Reference

## Timeline Entry Record

Each timeline entry is stored in `tx_maitimeline_entry` and carries six fields:

| Field | Type | Required | Notes |
|---|---|---|---|
| `title` | `input` (max 255) | ✔ | Record label in backend list; trimmed on save |
| `content` | `text` RTE (rows 15) | — | Rich-text body; uses RichText configuration `default` |
| `date` | `datetime` (date format) | ✔ | Calendar date of the event; used as alternative label when title is empty |
| `year` | `integer` (range 1900–2100) | — | Explicit year value for grouping; redundant with `date` but available for display and filtering without parsing the date |
| `image` | FAL file reference (max 1) | — | Allowed: all `common-image-types`; used as backend thumbnail |
| `categories` | `category` (manyToMany) | — | Links to `sys_category` — no custom category table |

Records are sorted by `date DESC` by default (most-recent event first).

## sys_category Integration

`mai_timeline` follows the project-wide architecture rule: **no custom category table**.

- The `categories` relation uses TYPO3's built-in `CategoryConfig` (TCA field type `category`),
  which maps to the shared `sys_category` table.
- Relations are stored in `sys_category_record_mm` (TYPO3 core table — no custom MM table).
- The same `sys_category` tree is shared with `mai_news`, `mai_gallery`, `mai_faq`, and
  `mai_testimonials`, enabling a single category hierarchy across all record types.

## Extbase Plugin: `mai_timeline_list`

`mai_timeline` ships a fully registered Extbase plugin that renders `tx_maitimeline_entry`
records on the frontend.

### Plugin Registration

Registered in `ext_localconf.php` as:

```
Extension key : MaiTimeline
Plugin name   : TimelineList
Controller    : TimelineController → list action
```

Editors insert it as a standard TYPO3 plugin content element (`CType = list`,
`list_type = maitimeline_timelinelist`).

### `TimelineController::listAction()`

The single action reads three FlexForm settings and passes a filtered result set to the
Fluid view:

| FlexForm setting | Type | Behaviour |
|---|---|---|
| `settings.storagePid` | page group (0–1 item) | Restricts the Extbase query to the selected storage page. 0 = respect TypoScript `persistence.storagePid`. |
| `settings.categoryUid` | category (manyToOne) | Filters entries via `EntryRepository::findByCategoryUid()`. 0 = all categories. |
| `settings.limit` | integer | Caps result count via `setLimit()`. 0 = no limit. |

The view receives a single `entries` variable — a `QueryResultInterface<Entry>` ordered by
`date DESC`.

### `EntryRepository`

Two public query methods:

- `findByCategoryUid(int $categoryUid, int $limit = 0): QueryResultInterface<Entry>` —
  filters by `categories` MM relation; applies optional limit.
- `findAllWithLimit(int $limit = 0): QueryResultInterface<Entry>` — full set with optional
  limit.

Both inherit `defaultOrderings = ['date' => ORDER_DESCENDING]`.

## Relationship to `mai_theme` Timeline Organism

`mai_theme` ships a `maispace_timeline` content element that uses a **different** data model:
`tx_maitheme_timeline_item`, an inline sub-table on `tt_content`. Those items carry
`event_date` (free-text string), `title`, `description`, and `image`. A
`DatabaseQueryProcessor` fetches `tx_maitheme_timeline_item` rows and passes them as
`items` to `Timeline.html`, which renders a `<section class="mai-timeline">` → `<ol>` →
`<li class="mai-timeline__item">` structure.

`tx_maitimeline_entry` and `tx_maitheme_timeline_item` are **independent tables** serving
different use cases:

| Aspect | `tx_maitheme_timeline_item` | `tx_maitimeline_entry` |
|---|---|---|
| Ownership | Inline to a `tt_content` record | Standalone records on a storage page |
| Date field | `event_date` — free-text string | `date` — TYPO3 date-picker (integer timestamp) |
| Year field | None | `year` — integer for grouping |
| Categories | None | `sys_category` (manyToMany) |
| Content field | `description` RTE | `content` RTE |
| Use case | Embedded, page-specific timelines | Shared historical records across pages |

## Database Tables

### `tx_maitimeline_entry`

The table is auto-generated by TYPO3 from the TCA definition (no `ext_tables.sql`).

| Column | SQL type | Notes |
|---|---|---|
| `uid` | `int(11)` | Auto-increment primary key |
| `pid` | `int(11)` | Storage page |
| `title` | `varchar(255)` | Record label; required |
| `content` | `mediumtext` | RTE body content |
| `date` | `int(11)` | Unix timestamp (date only — time component is midnight) |
| `year` | `int(11)` | Explicit year integer (1900–2100); defaults to 0 |
| `image` | `int(11)` | FAL reference count; actual files in `sys_file_reference` |
| `categories` | `int(11)` | MM counter (TYPO3 standard); relations in `sys_category_record_mm` |
| Standard enableFields | — | `hidden`, `deleted`, `starttime`, `endtime`, `sys_language_uid`, … |

The default backend sort order is `date DESC`.

## Architecture Constraints

- **No custom category table.** Category taxonomy is handled exclusively via `sys_category`.
  Never add a `tx_maitimeline_category` table or a custom MM join.
- **No SCSS.** Styling is owned exclusively by `mai_assets` (compiled SCSS) and `mai_theme`
  (organism-level styles in `_timeline.scss`). Never add `scssphp/scssphp` as a dependency.
- **No mail dispatch.** `mai_timeline` must never send email. All transactional email is
  routed through `mai_mail`.
- **Independent from `mai_theme` inline items.** `tx_maitimeline_entry` and
  `tx_maitheme_timeline_item` are separate tables. Do not merge them or create foreign-key
  dependencies between them.
- **FAL only.** Images are stored as `sys_file_reference` relations. Never store raw file
  paths in `tx_maitimeline_entry`.
